# 금언 저주 설계 제안

> 상태: 구현 전 검토 초안  
> 목적: 총괄 승인 후 별도 브랜치와 후속 PR에서 구현  
> 범위: 금언 저주 단일 항목의 의도, 언령 예산, 입력 차단 및 검증 기준  
> 공통 선택 규칙: [CURSE_SELECTION_PROPOSAL.md](./CURSE_SELECTION_PROPOSAL.md)

## 1. 제안 목적

`금언`은 긴 영창 문장에 의존하는 플레이어가 제한된 글자 안에서 의미와 창의성을 압축하도록 만드는 영창 카테고리 저주다.

금언방에서는 수동 영창 입력에 매우 작은 `언령 예산`을 적용한다. 플레이어는 긴 설명문을 그대로 사용할 수 없으며, 짧은 주문명·은유·합성어·압축 명령으로 원하는 마법을 표현해야 한다.

```text
긴 영창:
적에게 접근한 뒤 화염을 폭발시키고 뒤로 빠져라

압축 영창:
돌진후폭발
```

금언은 시퀀스 결과의 복잡성을 제한하지 않는다. 짧은 입력이 LLM에 의해 여러 behavior의 복합 시퀀스로 해석되는 것은 허용하며, 적은 언어로 풍부한 결과를 끌어내는 것이 핵심 대응법이다.

## 2. 저격할 플레이 습관

금언은 직전 런에서 긴 수동 영창 문장을 반복적으로 사용한 플레이어에게 높은 행동 점수를 부여한다.

단순 평균은 짧은 영창 몇 개로 쉽게 희석될 수 있으므로, 성공한 수동 영창들의 언령 비용을 정렬한 뒤 가장 긴 상위 25%의 평균을 원시 지표로 사용한다.

```text
금언 원시 지표
= 성공 수동 영창 중 상위 25% 언령 비용 평균
```

- 정상 발동한 직접 수동 영창 포함
- 감쇠 시전 포함
- 정상 발동한 cache/fallback 포함
- 시퀀스도 원본 입력 문장 기준으로 plan당 1회 포함
- 불발·차단·마나 소비 실패 제외
- 정령·각인·기본 공격·자동 공격 제외

수동 영창 표본이 부족한 런은 실제 상위 25% 값을 그대로 쓰지 않고 공통 중립값과 혼합하거나 중립값으로 처리한다. 최소 표본과 정규화 구간은 플레이테스트 데이터로 확정한다.

## 3. 언령 예산

금언방의 총 언령 예산은 `30`으로 둔다.

| 문자 종류 | 문자당 비용 |
|---|---:|
| 한글·한자 등 넓은 의미 문자 | `5` |
| 영문자·숫자 | `3` |
| 문장부호·기호 | `1` |
| 공백 | `0` |

따라서 다음 입력이 같은 한도를 가진다.

```text
한글 6자 × 5 = 30
영문 10자 × 3 = 30
```

혼합 입력은 문자별 비용을 합산한다.

```text
fire폭발
fire = 4 × 3 = 12
폭발 = 2 × 5 = 10
총 22 / 30 → 허용
```

```text
dash후화염폭발
dash = 4 × 3 = 12
후화염폭발 = 5 × 5 = 25
총 37 / 30 → 초과
```

공백은 영어 단어 구분과 한글 가독성을 위해 비용에서 제외한다. 문장부호와 기호는 완전히 무료로 두면 기호를 이용해 제한을 과도하게 우회할 수 있으므로 낮은 비용을 부여한다.

## 4. 글자 분류와 계산

사용자에게 보이는 문자 단위로 비용을 계산한다.

- 앞뒤 공백은 제거한 뒤 계산
- 내부 공백은 보존하되 비용 `0`
- `Intl.Segmenter`의 grapheme 단위를 우선 사용
- `Intl.Segmenter` 미지원 환경에서는 `Array.from(text)`로 폴백
- 한글 자모 조합 중간값으로 입력을 차단하거나 문자열을 잘라내지 않음

문자 종류 분류가 불가능한 grapheme은 기본적으로 기호 비용을 사용하되, 이모지처럼 하나의 의미 단위가 지나치게 낮은 비용으로 우회 수단이 되면 구현 단계에서 별도 비용을 조정한다.

비용 계산 함수는 입력 UI, 로컬 차단, 행동 기록 및 회귀 테스트가 공유하는 단일 출처로 둔다.

## 5. 입력과 차단

금언 예산을 초과해도 입력 자체를 막거나 문자열을 강제로 자르지 않는다. 플레이어가 전체 문장을 보며 직접 압축할 수 있어야 하고, 한글 IME 조합도 안전하게 유지해야 한다.

Enter로 영창을 확정할 때 언령 비용이 예산을 초과하면 로컬에서 즉시 차단한다.

- Gemini 호출 없음
- 마나 소비 없음
- 쿨다운과 입력 잠금 없음
- 주문 히스토리와 다음 런 행동 기록 없음
- 주문 반사 등 정상 발동 이후의 다른 영창 판정 없음
- 불발 SFX와 금언 전용 안내 표시
- 입력 내용은 유지하여 사용자가 수정 가능

안내 문구 초안:

```text
금언 · 더 짧은 언령으로 압축하세요
```

공백·기호만 있고 의미 문자가 없는 입력은 금언 전용 성공으로 처리하지 않고 기존 빈 영창 또는 불발 규칙을 따른다.

## 6. 입력 UI

일반 글자 수 표시 대신 현재 언령 비용을 실시간으로 표시한다.

```text
금언 22 / 30
한글 6자 · 영문 10자 상당
```

- 예산 이내: 기존 영창 UI 색상 유지
- 예산 초과: 비용 표시와 입력 테두리를 위험 색상으로 변경
- 예산 초과 상태에서 Enter: 짧은 흔들림 또는 금언 침식 효과
- 방 진입 시 제한과 대응법 안내
- 방 종료 시 일반 입력 한도와 기존 UI를 완전히 복구

입력 제한이 현재 방에만 적용된다는 사실과 남은 예산을 별도 설명 없이도 확인할 수 있어야 한다.

## 7. 시퀀스와 판정 결과

금언은 원본 입력 텍스트만 제한한다.

- 결과 plan의 시퀀스 수 제한 추가 없음
- sequence별 behavior 수 제한 추가 없음
- 원소·폼·효과 조합 제한 추가 없음
- 기존 로컬 시퀀스 검증과 정규화 규칙 유지

```text
돌진후폭발
→ move → fire nova → retreat 시퀀스로 해석 가능
```

짧은 입력이 단일 행동으로 판정될지 복합 시퀀스로 판정될지는 기존 LLM 스키마와 검증기의 책임이다. 금언은 결과를 임의로 복잡하게 만들지 않는다.

## 8. 짧은 입력 판정 품질 위험

금언의 가장 큰 위험은 제한 자체의 난도가 아니라, 6자 이하 한글 영창에서 LLM이 사용자의 의도를 충분히 해석하지 못하는 상황이다.

구현 후 다음 지표를 별도로 계측한다.

- 예산 이내 한글·영문 영창의 정상 발동률
- 단일 영창과 시퀀스 판정 비율
- cache/fallback 발생률
- 평균 및 상위 판정 지연
- 플레이테스트에서 보고된 의도 불일치 사례
- 동일한 짧은 영창의 결과 안정성

판정 성공률이 낮으면 언령 예산을 즉시 늘리기보다 Worker 프롬프트에 짧은 금언형 입력도 의미를 적극적으로 확장 해석하도록 지침을 추가하는 방안을 먼저 검토한다.

다만 짧은 입력을 근거 없이 과도한 복합 시퀀스로 확장하면 사용자 의도와 멀어질 수 있으므로, 정상 발동률과 의도 일치율을 함께 본다.

## 9. 기존 시스템과의 상호작용

- 금언 차단은 Gemini 호출과 마나 소비보다 먼저 판정
- 침묵 등 다른 입력 차단과 동시에 존재하지 않더라도 공통 차단 순서를 명확하게 유지
- 정상 발동한 감쇠·cache·fallback 영창은 금언 행동 기록에 포함
- 반복 페널티와 다양성 보너스는 금언을 통과해 실제 발동한 뒤 기존대로 적용
- 시퀀스는 원본 입력 언령 비용 1회만 기록
- 자동 공격·정령·각인은 금언 예산과 행동 기록에서 제외
- 보상·방 전환·런 종료 중에는 금언 UI와 제한 비활성화

## 10. 플레이테스트 항목

1. 한글 6자·영문 10자 상당 제한이 실제로 영창 습관을 바꾸는가
2. 제한이 단순 불편이 아니라 의미 압축과 창의성의 퍼즐로 받아들여지는가
3. 언령 비용과 남은 예산을 입력 중 즉시 이해할 수 있는가
4. 한글 IME 조합과 수정·삭제가 정상 작동하는가
5. 혼합 언어 및 문장부호 비용으로 과도한 우회가 발생하지 않는가
6. 6자 이하 한글 입력의 정상 발동률과 의도 일치율이 수용 가능한가
7. 짧은 입력에서도 시퀀스 판정이 필요한 경우 정상 생성되는가
8. 초과 차단이 Gemini·마나·쿨다운·히스토리를 소비하지 않는가
9. 방 종료 후 일반 입력 UI와 한도가 완전히 복구되는가
10. 상위 25% 언령 비용 가중치가 긴 영창 의존 습관을 제대로 구분하는가

## 11. 승인 요청 사항

1. 금언을 긴 수동 영창 문장을 저격하는 `casting` 카테고리 저주로 추가
2. 언령 예산 `30`과 문자별 비용 `한글·한자 5 / 영문·숫자 3 / 기호 1 / 공백 0` 적용
3. 예산 초과 입력은 허용하되 Enter 확정 시 Gemini 호출 전 로컬 차단
4. 차단 시 마나·쿨다운·히스토리 및 후속 영창 판정을 소비하지 않음
5. 결과 시퀀스와 behavior 복잡도에는 별도 제한을 추가하지 않음
6. 직전 런 성공 수동 영창 중 상위 25% 언령 비용 평균을 후보 원시 지표로 사용
7. 짧은 입력의 정상 판정률·시퀀스 비율·fallback·의도 일치율을 구현 후 별도 계측
8. 판정 품질 문제가 확인되면 예산 완화보다 Worker의 짧은 입력 해석 지침을 먼저 검토

승인 전에는 구현에 착수하지 않는다. 승인 과정에서 변경된 항목을 이 문서에 반영한 뒤 별도 브랜치에서 구현한다.
