> **제출물 ⑤** · NHN 2026 게임 AI 해커톤
> 원천 문서: [ROLES.md](ROLES.md)(2026-07-14 배정). 이 문서는 **실제 수행 내역**을 기준으로 작성했다.

| 역할 | 담당 | 한 줄 |
|---|---|---|

## R1 — 게임 코어 (Game Core) · 이도원

> 판정 결과를 **실제로 보이고 만져지는 전투**로 만드는 층. 머지 PR 22건.

| 영역 | 실제 산출물 |
|---|---|
| **전투 코어** | Phaser 씬·런 루프, 플레이어 이동·기본공격, 적 6종 행동, 웨이브 스포너, 충돌·피해 판정 |
| **주문 폼 렌더러** | `bolt`·`zone`·`rain`·`chain`·`cage`·`wall`·`orbit`·`nova` — 판정이 내놓는 12종 형태를 실제 전투 판정과 함께 구현 |
| **런 구조 v2·보스** | 2스테이지 6전투 5보상 구조, 스테이지 보스·기억 보스 패턴, 이중 저항 소비 |
| **게임필** | 히트스톱·카메라 셰이크·넉백, 전투 피드백·HUD 통합 폴리싱 |
| **에셋**(Phase 3~, 본인 희망) | AI 생성 사운드(SFX·BGM)·보스전 BGM, 타이틀 화면·내장 웹폰트·파비콘, 방별 배경 색조 |
| **프로토타입** | 능동형 마나 회복 C안(채택되어 시전 경제의 기반이 됨), nova 목표지점 투사체 |

**대표 기여** — *능동형 마나 C안*: 시전 경제 3개 후보 중 이도원이 플레이 가능한 프로토타입으로 만든 안이 팀 채택되어, **"적을 처치해 마나를 얻는다 = 영창하려면 싸워야 한다"**는 현재 코어 루프가 됐다.

## R2 — AI 시스템 (AI Systems) · 임재윤

> 자유 문장을 **구조화된 주문으로 바꾸는 층**과 그 인프라. 머지 PR 19건.

| 영역 | 실제 산출물 |
|---|---|
| **판정 프롬프트** | 의미 우선 판정 규칙(`meaning-v2.4`) — 다국어·비유 해석, fizzle은 키보드 매시만, 언어 중립 위력 |
| **LLM 인프라** | Cloudflare Worker 프록시 — API 키 은닉, 레이트리밋(15 RPM), CORS, **프롬프트 서버측 고정** |
| **폴백 체인** | localStorage 캐시 · 2.5초 타임아웃 · MockJudge 폴백 — *판정이 실패해도 게임은 멈추지 않는다* |
| **기억하는 보스** | 주문 히스토리 분석 → 원소 내성·카운터, 런 간 기억, LLM 도발 대사 생성 |
| **소환 행동 DSL** | `"지그재그로 돌진"` 같은 묘사를 **움직임 프로그램**으로 설계하는 프롬프트 — 이 게임의 최전선 기능 |
| **밸런스 리프레임** | 반복 억제를 페널티(채찍)에서 **다양성 보너스(당근)**로 전환 |
| **모델 안정화** | `-latest` 별칭 드리프트 사고 대응 → `gemini-3.5-flash-lite` 명시 핀 |
| **제출물 오너** | **④ AI 활용 기술 문서** |

**대표 기여** — *소환 행동 DSL*: 프롬프트만으로 LLM이 소환수 움직임을 설계하게 만들었고, **프롬프트 예시와 겹치지 않는 신규 묘사 20개 중 20개**가 유효한 프로그램으로 나왔다. "AI가 메뉴에서 고르는 게 아니라 행동을 설계한다"는 이 게임의 핵심 주장을 성립시킨 작업.

## R3 — 콘텐츠·UX·제출물 + 총괄 · 박재현

> 두 층을 **게임으로 붙이고**, 방향과 범위를 정하는 층. 머지 PR 64건.

| 영역 | 실제 산출물 |
|---|---|
| **성장 시스템** | 각인(자동 시전)·정령 계약·진화/융합, 보상 3택, 빌드 아키타입(소환사 vs 영창가) |
| **표현력 확장** | 자기 강화(가속·무적·돌진), 상태이상 4종, 소환 변형(분신·포탑·군체), 소환 행동 DSL 클라이언트 실행기 |
| **시전 경제** | 영창 환류, 감쇠 시전, 요금 가시화 — *마나가 없어 아무것도 못 하는 시간*을 없앤 설계 |
| **연출** | 영창 슬로모션·주문명 각인 타이포, 원소 친화 VFX 3단계 격상, 성장 피드백 |
| **아트 파이프라인** | AI 생성 스프라이트·배경의 누끼·워터마크 제거, 재질/발광 2층 렌더 분리 |
| **품질 방어** | 회귀 스크립트 34종 설계, 프로덕션 빌드 검증, 밸런스 계측(피해 귀속 원장) |
| **총괄** | 페이즈 지시, 게이트 결정(근거 실측), PR 리뷰·머지 전량, 이슈 배분, 일정 관리 |
| **제출물 오너** | **② 플레이 영상 · ③ 게임 소개 · ⑤ 역할 기술서** |

**대표 기여** — *게이트 결정*: 트랙 B 신규 기능(소환 행동 DSL 등)의 컷/승격을 **감으로 정하지 않고 실측으로 정했다**. 수율 20/20, 판정 폴백 0/10, 오토 화력 비중 실런 2회 측정 — 근거를 문서에 남기고 팀에 전파해 회의 없이 결론을 냈다.

## 협업 방식

- **PR 기반 협업**: main 직접 push·self-merge 금지. 전원 브랜치 작업 → 총괄 리뷰 후 머지 (머지된 PR 105건 전부 이 경로)
- **계약 우선(contract-first)**: 역할 경계의 인터페이스(판정 JSON 스키마, 보상 `RewardKind`, 보스 기억 계약)를 **먼저 커밋해 고정**하고 그 위에서 병렬 작업 — 3인이 서로 다른 에이전트를 동시에 몰 수 있었던 핵심
- **회귀로 방어**: 순수 로직을 회귀 스크립트 34종으로 고정. 다른 사람(또는 다른 에이전트)의 변경이 기존 계약을 깨면 즉시 드러난다
- **AI 사용 즉시 기록**: 작업 직후 [AI_USAGE_LOG.md](AI_USAGE_LOG.md)에 1줄씩 (누적 73건 — 제출물 ④의 원천 데이터)
- **인수인계 문서화**: 기계·AI 세션이 바뀌어도 이어갈 수 있도록 [HANDOFF.md](../HANDOFF.md)에 환경·함정·워크플로 고정

### 역할 간 인터페이스 계약

| 경계 | 계약 |
|---|---|
| R1 ↔ R2 | 판정 JSON 스키마(`SpellSpec`·`SpellJudgement` v2) — 변경 시 양측 합의 |
| R1 ↔ R3 | 보상 계약(`RewardKind`·`RunController` 이벤트), 씬 전환 |
| R2 ↔ R3 | 영창 UX 타이밍(판정 대기 연출), 보스 대사 포맷, 소환 행동 DSL 스키마 |

> 제출물 ①(빌드+소스)은 **전원 공동 책임** — 배포 파이프라인은 R3가 관리.
