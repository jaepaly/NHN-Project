# Phase 2 작업 지시 — 코어 런 확장

> 기간: **즉시 착수 ~ 2026-07-27**  
> 공식 W2: 7/21~7/27  
> 목표: **"방을 클리어하고 보상을 골라 다음 방으로 이어지는 한 런의 뼈대를 완성한다."**

## 1. 통합 완료 기준

아래 조건을 모두 만족하면 Phase 2를 완료한다.

- [ ] 새 런 시작 → 방 1의 3웨이브 격퇴 → 3택 1 보상 → 방 2 진입·클리어가 한 흐름으로 이어진다.
- [ ] 선택한 보상이 HP·마나·원소 친화 중 하나의 실제 전투 수치에 반영된다.
- [ ] 주문 폼 6종(`bolt`, `nova`, `beam`, `wave`, 신규 2종)이 시각·판정 면에서 구분된다.
- [ ] 성공한 주문이 런 히스토리에 기록되고, 동일 주문 반복 시 패널티가 HUD나 주문 메타에 보인다.
- [ ] 최근 주문 기록으로 최다 원소·최다 폼을 계산하는 보스 기억용 요약 계약이 준비된다.
- [ ] `라이트닝 스톰`, `숲의 분노`, `배고프다`를 의미에 맞게 발동하고 `ㅁㄴㅇㄹ`·금칙 입력을 각각 불발·차단한다.
- [ ] 회복·보호 주문이 적 피해가 아니라 플레이어 상태에 실제 반영된다.
- [ ] 로컬 Mock 모드와 라이브 Gemini 모드에서 각각 2개 방 흐름을 검증한다.
- [ ] `npm run build` 통과, 주요 브라우저 콘솔 오류 없음, Pages 배포 성공.

## 2. 공통 인터페이스 계약

1. Phase 2 첫 계약 PR에서 `SpellSpec v1`을 `SpellJudgement v2`로 전환한다. 이후 필드와 enum은 R1·R2 합의 없이 변경하지 않는다.
2. 주문 판정과 전투 적용을 분리한다. Gemini 출력값을 직접 저장하지 말고 검증된 `SpellSpec`만 히스토리에 기록한다.
3. 보상 선택은 추가 LLM 호출 없이 결정론적으로 처리한다.
4. 전투 반복 개발은 `VITE_JUDGE_MOCK=1`을 기본으로 하며, 실제 Gemini 품질 테스트만 공용 프록시를 쓴다.
5. 새 공유 타입은 역할 소유 폴더에 두고, 다른 역할은 공개 메서드·이벤트만 사용한다.
6. 씬·상태 계약 변경은 PR 본문에 **R1↔R2**, **R1↔R3** 영향 범위를 명시한다.

권장 런 상태 최소 형태:

```ts
interface RunState {
  roomIndex: number;
  maxRooms: number;
  rewards: RewardId[];
  spellHistory: SpellHistoryEntry[];
  elementalAffinity: Partial<Record<SpellElement, number>>;
}
```

필드명은 구현 과정에서 조정할 수 있지만, `방 진행 / 보상 / 주문 기록 / 원소 친화` 네 책임은 분리하지 않는다.

### P0 게이트 — Spell Understanding v2

방·보상 기능보다 먼저 [SPELL_UNDERSTANDING_V2.md](SPELL_UNDERSTANDING_V2.md)의 판정 계약을 확정한다.
구현 진행과 역할별 체크리스트는 [Issue #10](https://github.com/jaepaly/NHN-Project/issues/10)에서 통합 추적한다.

- R2: 의미 분류 → `cast/fizzle/blocked` → 효과 의도 → 원소·폼 순서로 판정
- R1: `damage/heal/shield/buff/control/summon` 효과 디스패처와 엔진 수치 공식
- R3: 불발·금칙 경고, 회복 숫자, 보호막 HUD와 접근 가능한 색상 피드백
- 공통: v2 캐시 접두사와 고정 회귀 코퍼스 도입

이 게이트는 Phase 2의 **절대 컷 금지** 항목이다.

## 3. 담당자별 지시

### R1 — 이도원: 런·방·보상 적용 코어

우선순위 P0:

- [ ] 현재 방 클리어 뒤 즉시 종료되는 흐름을 `combat → reward → next room` 상태 전이로 확장한다.
- [ ] 한 런에 최소 2개 방을 구성하고 방마다 WaveManager를 초기화한다.
- [ ] 보상 효과 적용 API를 만든다.
  - 최대 HP 증가 + 즉시 일부 회복
  - 최대 마나 증가 + 즉시 일부 회복
  - 특정 원소 위력 증가
- [ ] R2의 주문 히스토리 기록 API를 **마나 지불 후 실제 발동이 확정된 시점**에 호출한다.
- [ ] `effect`별 적용기를 분리하고 `heal`, `shield`, `buff`가 적 피해 콜백을 호출하지 않도록 한다.

우선순위 P1:

- [ ] 렌더러에 시각적으로 구분되는 폼 2종을 추가해 총 6종을 만든다. 권장: `zone`, `rain`.
- [ ] 신규 폼의 다중 적 적중과 수명 종료 시 오브젝트 정리를 확인한다.

R1 완료 기준:

- 2개 방과 보상 효과가 UI 없이도 상태·로그 기준으로 연결된다.
- 신규 폼을 포함한 6종이 최소 1회씩 적에게 피해를 준다.
- 전투 코어 외 DOM/CSS는 수정하지 않는다. 필요한 UI 이벤트는 R3에 계약으로 전달한다.

### R2 — 임재윤: 주문 히스토리·반복 패널티·기억 요약

우선순위 P0:

- [ ] `SpellJudgement v2` 타입·검증·프록시 JSON 스키마를 구현한다.
- [ ] 프롬프트를 `의미/언어 정규화 → disposition → effect/target → element/form → power/cost` 순서로 개편한다.
- [ ] 캐시 접두사에 스키마·프롬프트 버전을 넣고 v1 캐시를 무시한다.
- [ ] MockJudge에 공통 외래어·자연·회복 의도 최소 사전을 추가하되, 의미 없는 입력은 `fizzle`로 유지한다.
- [ ] 검증된 주문만 받는 런 단위 `SpellHistory` 모듈을 만든다.
- [ ] 최소 기록 필드: 원문, 주문명, 주·보조 원소, 폼, power, cost, 판정 소스, 발동 시각.
- [ ] 동일 정규화 문장 반복 시 엔진이 적용할 패널티 값을 계산한다.
  - 1회 재사용부터 기본 `power × 0.8`
  - 누적 규칙과 최솟값은 테스트로 고정
  - Gemini 프롬프트가 아니라 로컬 코드가 강제
- [ ] R1이 호출할 기록·조회 API와 R3가 읽을 표시용 요약 타입을 제공한다.

우선순위 P1:

- [ ] 최근 기록으로 `dominantElement`, `dominantForm`, `recentSpellNames`를 계산하는 `BossMemoryProfile` 초안을 만든다.
- [ ] 5티어별 5개, 최대 25개 입력으로 1차 품질 스냅샷을 남긴다.
  - 결과 JSON, latency, source, 티어 적합 여부 기록
  - 같은 입력 재검증은 캐시 사용
  - 대량 연속 요청 금지

R2 완료 기준:

- 히스토리·반복 패널티·요약 계산이 판정기와 독립된 단위 테스트 또는 재현 스크립트로 검증된다.
- 프록시 장애에서도 Mock 폴백 주문이 동일한 기록 계약을 따른다.
- 실제 보스 전투·대사 호출은 구현하지 않고 Phase 3용 계약까지만 만든다.

### R3 — jaepaly: 보상 UX·방 진행 연출·통합 QA

우선순위 P0:

- [ ] 방 클리어 시 3개 보상 카드를 표시하고 마우스 또는 `1/2/3`으로 하나를 선택하게 한다.
- [ ] 각 카드에 효과명·수치·영향 대상을 명확히 표시한다.
- [ ] 선택 전에는 다음 방이 시작되지 않으며, 선택 후 HUD 수치가 즉시 갱신된다.
- [ ] HUD에 `ROOM 1/2`, 현재 보상 또는 원소 친화 요약을 추가한다.
- [ ] 방 전환 시 0.5~1초 페이드·문구 연출을 넣되 전투 상태 로직은 R1 API를 사용한다.
- [ ] `fizzle`은 "마력이 형태를 이루지 못했다", `blocked`는 명확한 입력 경고를 표시한다.
- [ ] `heal`은 녹색 회복 숫자, `shield`는 별도 보호막 게이지로 공격 주문과 구분한다.

우선순위 P1:

- [ ] 반복 패널티 발생 시 `REPEAT -20%`처럼 원인을 플레이어에게 표시한다.
- [ ] 보상 카드·방 전환·신규 폼 6종의 색상과 텍스트 대비를 검수한다.
- [ ] Phase 2 통합 빌드를 Pages에 배포하고 30초 내부 검증 영상을 남긴다.

R3 완료 기준:

- 키보드만으로 영창과 보상 선택을 모두 진행할 수 있다.
- 960×640 기준 HUD가 겹치지 않고, FIT 스케일에서 보상 카드가 화면 밖으로 나가지 않는다.
- 라이브 Gemini 요청은 최종 통합 확인에 필요한 최소 횟수만 사용한다.

## 4. 권장 병합 순서

1. **Spell Understanding v2 계약**: R2 타입·검증·프롬프트 → R1 효과 적용 → R3 피드백 순서로 병합
2. **런 계약 확정**: R1 런 상태/이벤트 + R2 히스토리 공개 API 공유
3. **병렬 구현**: R1 방 진행·보상 적용 / R2 히스토리·패널티
4. **R3 UI 연결**: 확정된 R1 이벤트와 상태만 사용
5. **렌더러 2종 추가**: 공용 `render/` 충돌을 피하도록 R1 PR 뒤에 병합
6. **통합 QA**: 고정 의미 코퍼스 → Mock 완주 → 빌드 → Pages → Gemini 최소 검증

## 6. 스코프 컷 순서

일정이 밀리면 아래 순서로 줄인다.

1. 원소 친화 보상을 제거하고 HP·마나 2종 + 회복 변형으로 3카드를 구성
2. 신규 폼을 2종에서 1종으로 축소
3. 방 2의 웨이브 구성을 방 1 재사용으로 단순화
4. 5티어 품질 입력을 티어별 3개로 축소

**절대 컷 금지**: 의미 판정 v2와 불발/금칙 구분, 회복 효과 1종, 방 클리어 → 보상 선택 → 다음 방, 주문 히스토리, 반복 패널티의 플레이어 피드백.
